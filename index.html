Data Quality
7 validation rules run on every reading: null check, negative power, temperature range, wind range, power capacity, duplicate timestamps, sparse data. Each record scored as OK, DEGRADED, or SUSPECT.
Anomaly Detection
4 SQL detectors identify HIGH_WIND_ZERO_POWER, DATA_GAP, TEMPERATURE_SPIKE, and SUSPECT_DATA. Anomalies clustered into actionable incidents per turbine.
7 Validation Rules
  • Null Value Check (WARNING) — Detects missing or null sensor readings in active_power, wind_speed, and temperature fields. Flags records where critical measurements are absent.
  • Negative Power (WARNING) — Flags active_power < 0. Small negatives may indicate parasitic load; large negatives suggest sensor malfunction.
  • Temperature Range (ERROR) — Validates gearbox and bearing temps within –40°C to 150°C physical bounds. Outside this range indicates sensor failure.
  • Wind Speed Range (WARNING) — Checks wind_speed between 0 and 60 m/s. Values beyond cut-out speed (25 m/s) flagged for review.
  • Power Exceeds Capacity (WARNING) — Cross-references active_power against turbine_models.rated_power. Power exceeding 110% of rated capacity signals data issue.
  • Duplicate Timestamps (INFO) — Identifies duplicate (turbine_id, timestamp) pairs. Keeps latest record, logs duplicates for audit trail.
  • Sparse Data Gaps (WARNING) — Detects periods with < 50% expected readings. Indicates communication failures or unplanned turbine downtime.
OK — 0% errors DEGRADED — 0–50% errors SUSPECT — >50% errors
4 SQL-Based Anomaly Detectors
  • HIGH_WIND_ZERO_POWER
    Detects when wind_speed > 3.5 m/s but active_power = 0. Indicates potential turbine failure, brake engagement, or grid disconnection. Uses a 15-minute rolling window query joining sensor_readings with turbine status.
  • DATA_GAP
    Identifies gaps > 30 minutes in sensor readings using LAG() window function over timestamps. Signals SCADA communication failures, network issues, or unplanned turbine shutdowns.
  • TEMPERATURE_SPIKE
    Flags gearbox or bearing temperatures exceeding 80°C, using window functions to detect > 15°C change within 30 minutes. Critical for preventing mechanical failure.
  • SUSPECT_DATA
    Promotes any reading with quality_flag = SUSPECT (> 50% validation errors) to an anomaly. Catches systemic data quality issues that individual rules might miss when combined.

Database Design

Redis 7 — Caching & Rate Limiting
  • AI Report Caching
    Second-level cache for DeepSeek LLM responses. Redis TTL of 5 minutes sits between the 30-minute database cache and the client. Reduces AI API calls by ~95% during traffic spikes.
  • Rate Limiting Counters
    Token bucket implementation using Redis INCR and EXPIRE. Each API key gets a configurable request quota tracked as Redis keys with automatic expiration.
  • Session & Config Store
    Stores transient application state and configuration values. Spring Boot auto-configured with RedisTemplate and Jackson serialization (JavaTimeModule for LocalDateTime).
  • Infrastructure
    Port 6379, running in Docker with health check via redis-cli ping. Connected to Spring Boot API via dashed dependency (optional — app degrades gracefully without Redis).

Resilience & Monitoring
AI Resilience Patterns
  • Circuit Breaker — 5-failure threshold opens circuit for 60s recovery. Prevents cascading failures when DeepSeek API is down.
  • 3-Layer Caching — DB cache (ai_report_cache table, 30min TTL) → Redis cache (5min TTL) → Client-side cache. Reduces API calls by ~95%.
  • Rate Limiting — Token bucket algorithm per API key, backed by Redis counters. Prevents abuse and controls costs.
  • Graceful Degradation — When AI service is unavailable, returns cached reports or "analysis unavailable" instead of failing.
Observability Stack
  • Prometheus — Scrapes Spring Boot Actuator metrics every 15s: JVM memory, HTTP request latency, HikariCP connections, cache hit rates.
  • Grafana — Pre-configured dashboards for API performance, database connections, and pipeline run status.
  • Spring Boot Actuator — /health, /metrics, /info endpoints. Health checks for DB, Redis, and disk space.
  • Correlation IDs — CorrelationIdFilter assigns UUID to every request. Propagated through logs for end-to-end tracing.
